# SituationMonitor

Go service that aggregates news (RSS), X/social data (Grok sweeps via OpenRouter), and market quotes into a local dashboard served over Tailscale. Includes a knowledge graph for entity extraction, item clustering, and situation tracking.

## Architecture

- Single-process Go binary with independent worker goroutines (RSS, sweep, translate, market, extract)
- SQLite database at `./data/situation.db`
- HTTP server at `127.0.0.1:8080` (configurable via `LISTEN_ADDR`)
- All config via environment variables (see `internal/config/config.go`)

## Knowledge Graph

The extract worker (`internal/extract/worker.go`) runs a 4-pass pipeline on each tick:

1. **NER extraction** — Ollama (nemotron-mini) extracts PERSON/ORG/PLACE/TOPIC entities from item titles and summaries
2. **Clustering** — items sharing 2+ entities are grouped via `cluster_key`
3. **Situation creation** — clusters with 4+ items are auto-named via Ollama and tracked as situations
4. **Normalization** — merges duplicate entities (Trump/Donald Trump, US/U.S./United States), fixes country kinds to PLACE, auto-links sub-situations to parents via entity overlap

Entity merge rules live in `internal/store/merge.go`. Add new name variants or country entries there.

## Daily Briefings

Daily briefing pages are generated by Claude via the `/daily-briefing` skill and written as YAML files to `data/pages/YYYY-MM-DD.yaml`. The Go server renders these at `/daily/{date}` with an index at `/daily/`.

The skill runs a Step 0 knowledge graph curation (merge duplicate situations, fix hierarchy, clean junk entities) before gathering data, doing web searches, and producing the briefing.

**Automated schedule:** A systemd user timer runs the briefing at 6:57 AM CT daily:
- `~/.config/systemd/user/daily-briefing.timer` / `daily-briefing.service`
- `systemctl --user status daily-briefing.timer` to check
- `systemctl --user start daily-briefing.service` to trigger manually
- `journalctl --user -u daily-briefing.service` for logs

## Key paths

- `cmd/situation-monitor/main.go` — entry point
- `internal/httpserver/` — HTTP routes and templates
- `internal/httpserver/intel.go` — knowledge graph API endpoints (brief, situations, entities)
- `internal/httpserver/manage.go` — management API endpoints (merge, rename, delete, set parent)
- `internal/store/` — database access layer
- `internal/store/entity.go` — entity CRUD and related-item queries
- `internal/store/situation.go` — situation CRUD and hierarchy
- `internal/store/merge.go` — entity normalization rules and merge logic
- `internal/store/ner.go` — NER worker queries (unextracted items, clustering, batch ops)
- `internal/store/manage.go` — management operations (rename, delete, merge situations/entities)
- `internal/extract/worker.go` — 4-pass extract worker (NER → cluster → situations → normalize)
- `internal/ollama/ner.go` — entity extraction via Ollama
- `internal/ollama/summarize.go` — contextual briefing and situation naming via Ollama
- `internal/openrouter/` — Grok sweep client (X/social data collection)
- `config/brief.txt` — sweep focus areas
- `config/feeds.txt` — RSS feed URLs
- `data/pages/` — generated daily briefing YAML files
- `.claude/skills/daily-briefing.md` — Claude skill for generating briefings

## API endpoints

### Data APIs
- `GET /api/items?hours=&limit=&source=&min_u=` — news items
- `GET /api/markets` — market quotes
- `GET /api/sweeps?limit=` — Grok sweep results
- `GET /api/brief/{id}` — on-demand contextual synthesis for an item (calls Ollama)
- `GET /api/situations?status=&limit=&tree=true` — list situations (tree=true for hierarchy)
- `GET /api/situations/{slug}` — situation detail with items and children
- `GET /api/entities/{name}` — entity detail with recent items
- `GET /api/entities?kind=&limit=` — list top entities

### Management APIs
- `POST /api/entities/merge` — `{"from_id": N, "to_id": N}`
- `POST /api/entities/{id}/rename` — `{"name": "New Name"}`
- `DELETE /api/entities/{id}` — delete entity and links
- `POST /api/situations/merge` — `{"from_id": N, "to_id": N}`
- `POST /api/situations/{id}/parent` — `{"parent_id": N}` or `{"parent_id": null}`
- `POST /api/situations/{id}/rename` — `{"name": "New Name"}`
- `POST /api/situations/{id}/status` — `{"status": "active|resolved|watching"}`
- `DELETE /api/situations/{id}` — delete situation

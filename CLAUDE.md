# SituationMonitor

Go service that aggregates news (RSS), X/social data (Grok sweeps via OpenRouter), and market quotes into a local dashboard served over Tailscale. Includes a knowledge graph for entity extraction, item clustering, and situation tracking.

## Architecture

- Single-process Go binary with independent worker goroutines (RSS, sweep, translate, market, extract)
- SQLite database at `./data/situation.db`
- HTTP server at `127.0.0.1:8080` (configurable via `LISTEN_ADDR`)
- All config via environment variables (see `internal/config/config.go`)

## Knowledge Graph

The extract worker (`internal/extract/worker.go`) runs a 4-pass pipeline on each tick:

1. **NER extraction** ‚Äî Ollama (nemotron-mini) extracts PERSON/ORG/PLACE/TOPIC entities from item titles and summaries
2. **Clustering** ‚Äî items sharing 2+ entities are grouped via `cluster_key`
3. **Situation creation** ‚Äî clusters with 4+ items are auto-named via Ollama and tracked as situations
4. **Normalization** ‚Äî merges duplicate entities (Trump/Donald Trump, US/U.S./United States), fixes country kinds to PLACE, auto-links sub-situations to parents via entity overlap

Entity merge rules live in `internal/store/merge.go`. Add new name variants or country entries there.

## Daily Briefings

Daily briefing pages are generated by Claude via the `/daily-briefing` skill and written as YAML files to `data/pages/YYYY-MM-DD.yaml`. The Go server renders these at `/daily/{date}` with an index at `/daily/`.

The skill runs a Step 0 knowledge graph curation (merge duplicate situations, fix hierarchy, clean junk entities) before gathering data, doing web searches, and producing the briefing.

**Automated schedule:** A systemd user timer runs the briefing at 6:57 AM CT daily:
- `~/.config/systemd/user/daily-briefing.timer` / `daily-briefing.service`
- `systemctl --user status daily-briefing.timer` to check
- `systemctl --user start daily-briefing.service` to trigger manually
- `journalctl --user -u daily-briefing.service` for logs

## Constraint-Based Alpha

Multi-domain analysis layer using Marko Papic's framework. Claude is the analyst ‚Äî the system provides storage, APIs, and UI for persisting and displaying constraint-based analysis across four domains.

**Core concept:** Forecast based on what actors *can actually do* (constraints), not what they *want* (preferences). Each domain has a Claude skill for performing analysis.

**Domains:**
- `geopolitics` ‚Äî `/geopolitical-alpha` ‚Äî political, economic, geopolitical, constitutional, time constraints
- `macro` ‚Äî `/macro-alpha` ‚Äî monetary, fiscal, labor, inflationary, time constraints
- `semiconductors` ‚Äî `/semiconductors-alpha` ‚Äî physical, capacity, regulatory, economic, competitive constraints
- `energy` ‚Äî `/energy-alpha` ‚Äî supply, demand, infrastructure, regulatory, geopolitical constraints

**Tables:** `constraints`, `net_assessments`, `assessment_constraints`, `probability_updates`, `alpha_calendar`, `data_streams` ‚Äî all domain-aware via `domain` column

**UI:** `/alpha` ‚Äî landing page with domain cards. `/alpha/{domain}` ‚Äî per-domain dashboard.

## API endpoints

### Data APIs
- `GET /api/items?hours=&limit=&source=&min_u=` ‚Äî news items
- `GET /api/markets` ‚Äî market quotes
- `GET /api/sweeps?limit=` ‚Äî Grok sweep results
- `GET /api/brief/{id}` ‚Äî on-demand contextual synthesis for an item (calls Ollama)
- `GET /api/situations?status=&limit=&tree=true` ‚Äî list situations (tree=true for hierarchy)
- `GET /api/situations/{slug}` ‚Äî situation detail with items and children
- `GET /api/entities/{name}` ‚Äî entity detail with recent items
- `GET /api/entities?kind=&limit=` ‚Äî list top entities

### Management APIs
- `POST /api/entities/merge` ‚Äî `{"from_id": N, "to_id": N}`
- `POST /api/entities/{id}/rename` ‚Äî `{"name": "New Name"}`
- `DELETE /api/entities/{id}` ‚Äî delete entity and links
- `POST /api/situations/merge` ‚Äî `{"from_id": N, "to_id": N}`
- `POST /api/situations/{id}/parent` ‚Äî `{"parent_id": N}` or `{"parent_id": null}`
- `POST /api/situations/{id}/rename` ‚Äî `{"name": "New Name"}`
- `POST /api/situations/{id}/status` ‚Äî `{"status": "active|resolved|watching"}`
- `DELETE /api/situations/{id}` ‚Äî delete situation

### Alpha APIs (all support `?domain=` filter)
- `GET /api/alpha/domains` ‚Äî domain summaries (assessment/constraint/event counts)
- `GET /api/alpha/dashboard?domain=` ‚Äî composite: active assessments + calendar + updates
- `GET /api/constraints?situation_id=&domain=&type=&region=&status=` ‚Äî list constraints
- `POST /api/constraints` ‚Äî create constraint
- `GET /api/constraints/{id}` ‚Äî get constraint
- `PUT /api/constraints/{id}` ‚Äî update constraint
- `DELETE /api/constraints/{id}` ‚Äî delete constraint
- `GET /api/assessments?situation_id=&domain=&lens=&status=` ‚Äî list assessments
- `POST /api/assessments` ‚Äî create assessment
- `GET /api/assessments/{id}` ‚Äî detail with constraints and update history
- `PUT /api/assessments/{id}` ‚Äî update assessment
- `DELETE /api/assessments/{id}` ‚Äî delete assessment
- `POST /api/assessments/{id}/update-probability` ‚Äî log Bayesian update
- `POST /api/assessments/{id}/constraints` ‚Äî link constraint to assessment
- `DELETE /api/assessments/{aid}/constraints/{cid}` ‚Äî unlink constraint
- `GET /api/calendar?from=&to=&domain=&region=&status=` ‚Äî list calendar events
- `POST /api/calendar` ‚Äî create calendar event
- `PUT /api/calendar/{id}` ‚Äî update calendar event
- `DELETE /api/calendar/{id}` ‚Äî delete calendar event
- `GET /api/data-streams?constraint_id=` ‚Äî list data streams
- `POST /api/data-streams` ‚Äî create data stream
- `PUT /api/data-streams/{id}` ‚Äî update data stream
- `DELETE /api/data-streams/{id}` ‚Äî delete data stream
